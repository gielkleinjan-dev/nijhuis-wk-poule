## Issue #1 — `profiles.paid` zelf-update

`authenticated` rol heeft kolom-UPDATE op alle kolommen van `profiles`.
RLS-policy `profiles update self` staat update toe waar `auth.uid() = id`.
Dus: een ingelogde user kan via PostgREST direct zijn eigen `paid` op `true` zetten.

De `is_admin` kolom is geen risico: de `is_admin()` postgres function leest de JWT-email tegen een hardcoded lijst, niet de kolom.

```sql
-- Vereist:

create policy "team_rank_snapshots read" on public.team_rank_snapshots
  for select to authenticated using (true);
-- Writes alleen via service_role (cron) — geen aparte policy nodig
```

## Issue #3 — `leaderboard` SECURITY DEFINER view

View draait met rechten van maker → omzeilt RLS van onderliggende tabellen.
Fix: view recreëren zonder SECURITY DEFINER, of `set (security_invoker = on)` zetten.

## Veilige bevindingen

- `admin_set_lock_at(timestamptz)`: `SECURITY DEFINER` MAAR met interne `is_admin()` check → safe.
- `cron_replace_user_points` / `cron_update_match_scores`: secret-parameter wordt intern gevalideerd tegen `cron_config.secret` → safe.
- `is_admin()` en `validate_invite_code()`: legitiem publiek aanroepbaar.
- App-code: geen secrets, Service Role uitsluitend server-side, admin-routes doen server-side `isAdmin()` check.
